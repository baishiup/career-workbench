# 列表页向量匹配(Embedding 方案 B)

_状态:📐 设计中,尚未落地_

> 背景:规则匹配(`packages/domain/src/job/match.ts`)已删除——静态别名词典无法处理大小写、
> 技术写法、技能范围/层级差异,把它表达成精确百分比是过度承诺。详情页匹配度已改由 `job_match`
> 大模型产出(见 [工作机会](./工作机会.md))。但 AI 报告是按需、单职位、有成本的,**列表页
> 跑不起**——所以列表匹配度暂时下线,留待本方案补回。

## 目标

给职位列表提供一个**便宜、本地、可全量**的语义匹配信号,替代被删的规则分。要能处理规则
匹配处理不了的情况:技术写法差异(React 18 ≈ React、K8s ≈ Kubernetes)、技能范围与层级
(「数据库优化经验」可支撑「PostgreSQL」要求)、中英混排。

## 边界(非目标)

- 不替代详情页的 AI 报告:详情页仍由 `job_match` 给出匹配度 + evidence/gaps/risks。
- 不在列表页实时调用任何大模型 / embedding API(成本会被列表的 N 条职位放大)。
- 不做自训练 / 微调——当前数据量和调用量下,预训练 embedding API 性价比远更高。

## 核心原则

**把成本挪到写入时,列表页只读缓存。** 匹配信号 = 两个预存向量的余弦相似度,是一次纯数学
运算(pgvector 的 `<=>` 算子),不碰任何模型。花算力的「文本 → 向量」只在 JD 入库、Profile
更新时各做一次。

## 数据与状态边界

### 模型选型

| 维度 | 选择 | 理由 |
| --- | --- | --- |
| 模型 | `text-embedding-3-small`(OpenAI,或同档多语 API) | 多语、支持中英混排,8191 token 输入 |
| 维度 | 1536(可降维到 512 省存储) | pgvector 列维度须与模型一致,换模型要重算全表 |
| 不用 Supabase 内置 `gte-small` | 否决 | 384 维、512 token、英文为主,对中文不可靠 |
| 不自托管 HuggingFace(bge-m3 等) | 暂不 | 只有 API 费随调用量变肉疼且需中文原生强时才值得 |

约束:

- **Token 上限:** 不 embed 原始全文,而是 embed**蒸馏文本**(结构化字段拼成短文本),既绕开
  长度限制、信号也更干净。JD 例:`"前端工程师 | React, TypeScript, Next.js | 5年 | 远程"`;
  Profile 例:`"资深前端 | React, TypeScript, Node | 6年 | 期望远程"`。
- **归一化:** OpenAI 输出已归一化;改自托管模型要存前 `normalize`,否则余弦不准。
- **语言:** 必须用多语模型,保证中文 Profile ↔ 英文 JD 跨语言对上。

### 数据流

1. **JD 入库 / 更新**(`job-parse` 或职位编辑保存)→ 生成蒸馏文本 → 调 embedding API →
   写 `job_descriptions.embedding`。
2. **Profile 更新**(保存 Profile)→ 生成蒸馏文本 → 调 embedding API → 写
   `profiles.embedding`。
3. **列表页查询:** SQL 直接算当前用户 Profile 向量与各 JD 向量的余弦距离,返回 `match_score`,
   不调任何模型。

### 数据模型(pgvector)

```sql
create extension if not exists vector;

alter table job_descriptions add column embedding vector(1536);
alter table profiles         add column embedding vector(1536);

-- 可选:列表量大时加近邻索引
create index on job_descriptions using ivfflat (embedding vector_cosine_ops);
```

列表查询(`<=>` 是余弦距离,`1 - 距离` 即相似度):

```sql
select j.id, j.title,
       round((1 - (j.embedding <=> p.embedding)) * 100)::int as match_score
from job_descriptions j
cross join (select embedding from profiles where user_id = auth.uid()) p
where j.is_active and j.embedding is not null and p.embedding is not null
order by j.embedding <=> p.embedding;
```

## 用户可见行为

- 列表页「匹配度」列展示 `match_score`(0–100)。沿用诚实表达:可只显示区间标签或弱化数字,
  避免再让用户盯着未校准的精确百分比。
- Profile / JD 没有向量时显示「—」或「完善 Profile」(沿用此前列表空状态)。

## 待决问题

- embedding API 供应商与 key 管理(复用现有 LLM provider 还是单独接)。
- 蒸馏文本的确切字段拼法,需与 `job_match` 输入对齐,避免两套口径。
- 向量写入触发点:edge function 同步写,还是异步补算(入库先不阻塞)。
- 列表 `match_score` 的展示粒度(精确数字 vs 区间标签)。

## 验收(落地步骤)

1. 开 `pgvector` 扩展,加 `embedding` 列 + migration。
2. 封装 embedding 调用(蒸馏文本 → 向量)于 shared 层。
3. JD 入库/编辑保存、Profile 保存处接入写向量。
4. 一次性回填存量 JD / Profile 的向量。
5. 列表页查询改为带 `match_score` 的 SQL,恢复「匹配度」列。
