# 功能规格 · 人机协作开发流程

这个文件夹记录的不只是"功能怎么做",更是**我和 AI 协作开发这个项目的方式**。

## 这套流程怎么运转

1. **拆分。** 我和 AI 先把整个系统拆成若干**模块**,每个模块写成一份 spec(就是本目录下的
   文件)——它定义这个模块要做什么、边界在哪、怎么算做完。
2. **一次一篇。** 后续每一轮开发会话,只挑**一份 spec** 推进;跨 spec 的依赖只记为前置条件,
   不顺手实现(这条规则也写在 [AGENTS.md](../AGENTS.md))。
3. **spec 即事实源。** 实现以 spec 为准;实现完成后回头更新 spec 的状态与边界,让文档和代码
   始终对齐。

所以这些 spec 既是**设计输入**,也是**协作留痕**——能看到每个模块是如何被想清楚、再被一步步
实现的。

## 标准 spec 结构

每份 spec 按下面的结构写(可按需增删小节,但顺序与核心小节保持一致):

```markdown
# <模块名>

*状态:✅ 已实现 / 🟡 部分实现 / 📐 设计中*

## 目标
这个模块做什么、为什么(用户价值 / 在闭环中的位置)。

## 边界
明确「做」与「不做」;跨 spec 依赖只记为前置条件,不在本 spec 实现。

## 用户可见行为
关键交互、界面行为、状态流转——用户实际看到和操作的。

## 数据与状态边界
涉及的表 / 字段 / 状态机;数据从哪来、到哪去、谁是事实源。

## 失败与降级
失败如何展示;如何降级(如缺 Dify → mock / 手动填写)。

## 验收
怎么算做完;如何验证(可运行的检查或手动路径)。

## 相关
关联的代码、[ADR](../docs/architecture.md#decisions)、其他 spec。
```

> 这个结构呼应一条核心要求:**每个功能都必须能回答**——用户可见行为是什么、数据与状态边界
> 在哪、失败状态怎么展示、如何验证。
