# 简历列表页操作完善

_状态:✅ 已实现_

## 目标

完善 `/resumes` 简历列表页的操作列。操作列固定显示 `查看` 和 `更多`,`更多` 下拉包含
`修改名称`、`更新到 profile`、`导出`、`删除`。

## 边界

- `导出` 为预留功能:点击只 toast 提示 `开发中`,不做实际文件生成。
- `更新到 profile` 复用已有 `apply-resume-to-profile` Edge Function——**不**新增 Edge
  Function,**不**修改数据库 schema。

## 用户可见行为

- **查看:** 读取当前简历详情并打开查看/编辑弹窗;读取中显示 loading,避免重复点击。
- **更多:** 下拉菜单展示 `修改名称`、`更新到 profile`、`导出`、`删除` 四个动作。
- **修改名称:** 弹窗内输入框默认填入当前名称;trim 后不能为空;保存后更新
  `resumes.title` 和 `updated_at`;成功后刷新列表行名称,若同一份简历的查看弹窗打开则同步
  标题。
- **更新到 profile:** 二次确认弹窗 → 确认后按钮 loading → 调 `apply-resume-to-profile`
  (读 `resumes.ai_parsed_draft_json`,重新归一化为 `ProfileDraft`,覆盖
  `profiles.profile_data`)→ 成功提示已更新 profile。
- **删除:** 删除前判断当前用户简历数量;只剩最后 1 份时不允许删除并提示
  `至少保留一份简历`;成功后从列表移除;若被删简历正在查看则关闭弹窗。

## 数据与状态边界

前端 API:

- `renameResume(resumeId, title) → ResumeListRow`:更新 `resumes.title` + `updated_at`,
  返回列表所需字段。
- `deleteResume(resumeId) → { id }`:先统计当前用户可见简历数量,`count <= 1` 抛错,否则
  删除对应 `resumes` 记录。
- 复用 `applyResumeToProfile(resumeId)`:不新增 Edge Function、不改 schema。

UI 状态:列表加载中保持现有 loading;查看详情中对应 `查看` 显示读取状态;修改名称/更新
profile 保存中按钮 loading 且禁止重复提交;删除中禁用当前行更多操作;错误统一用 toast +
页面已有错误模式展示。

## 失败与降级

- 任一写操作失败用 toast + 页面已有错误信息模式展示,不留半成品状态。
- 最后 1 份简历不可删,给出明确提示。

## 验收

构建检查:

```bash
pnpm --filter @career-workbench/web check
pnpm functions:check
pnpm --filter @career-workbench/web build
```

浏览器验证 `/resumes`:

- 操作列显示 `查看`、`更多`。
- 修改名称能真实写入 Supabase,刷新后仍保留。
- 更新到 profile 会二次确认,并真实覆盖 `profiles.profile_data`。
- 导出只提示 `开发中`。
- 多份简历时删除成功;最后 1 份不能删除。
- 控制台无相关 error/warn。
