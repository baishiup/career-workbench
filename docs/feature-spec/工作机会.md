# 工作机会

## 目标

工作机会由管理员导入，普通用户只浏览职位并基于职位生成 target job 简历。

## MVP 导入方式

管理员可以：

- 粘贴职位链接。
- 粘贴完整 JD 文本。
- 上传职位截图。
- 手动填写公司、岗位、地点、远程状态、技能标签和 JD。

## LinkedIn 处理策略

- MVP 不依赖服务端自动抓 LinkedIn。
- 首选手动粘贴 JD 文本或截图导入。
- 如果后续做浏览器辅助采集，只在用户/管理员本机已登录的当前页面读取可见 DOM 文本。
- 服务端不保存 cookies，不自动登录，不做绕过反爬。
- 抓取失败必须能降级到截图 OCR 或手动粘贴。

## 职位字段

- 来源平台。
- 原始链接。
- 公司。
- 岗位名称。
- 地点。
- 远程状态。
- 岗位类型。
- 年限要求。
- 必备技能。
- 加分技能。
- 职责描述。
- 任职要求。
- 薪资范围。
- 发布时间。
- 导入方式。
- 导入状态。

## 用户侧能力

- 查看职位列表。
- 查看职位详情。
- 查看结构化 JD。
- 触发匹配分析。
- 生成 target job 简历。

## Dify 工作流

职位相关 AI 能力优先通过 Dify Workflow 编排。

- `job_parse`：把管理员粘贴的 JD 文本或 OCR 文本转成结构化职位字段。
- `job_match`：对比 `ProfileVersion`、base `ResumeVersion` 和 `JobDescription`，输出匹配分、命中证据、缺口和风险表达。
- `resume_generate`：基于匹配报告生成 target job resume 草稿。

要求：

- 普通用户触发匹配和生成，但不能修改原始职位。
- Dify 输出必须写入 `MatchReport`、`ResumeVersion` 或待采纳 patch，不能只停留在 Dify 运行结果里。
- 每次匹配和生成都要保存 `AiRun` 与 Dify 外部运行引用。
- Dify 失败时职位详情仍可浏览，用户可以重试匹配；本地开发 fixture 只能用于验证 UI 和事件协议。

## 验收标准

- 普通用户不能上传职位。
- 管理员能创建、编辑、停用职位。
- 职位必须有导入来源和导入状态。
- LinkedIn 导入失败时能降级为手动文本或截图导入。
- 匹配分析和定制简历生成能关联对应 Dify workflow run。
