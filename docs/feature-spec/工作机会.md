# 工作机会

## 目标

工作机会由管理员导入，普通用户只浏览职位并基于职位生成 target job 简历。

## MVP 导入方式

管理员可以：

- 粘贴职位链接。
- 粘贴完整 JD 文本。
- 上传职位截图。
- 手动填写公司、岗位、地点、远程状态、技能标签和 JD。

## LinkedIn 处理策略

- MVP 不依赖服务端自动抓 LinkedIn。
- 首选手动粘贴 JD 文本或截图导入。
- 如果后续做浏览器辅助采集，只在用户/管理员本机已登录的当前页面读取可见 DOM 文本。
- 服务端不保存 cookies，不自动登录，不做绕过反爬。
- 抓取失败必须能降级到截图解析或手动粘贴。

## 职位字段

- 来源平台。
- 原始链接。
- 公司。
- 岗位名称。
- 地点。
- 远程状态。
- 岗位类型。
- 年限要求。
- 必备技能。
- 加分技能。
- 职责描述。
- 任职要求。
- 薪资范围。
- 发布时间。
- 导入方式。

## 用户侧能力

- 查看职位列表，含规则匹配分。
- 查看职位详情。
- 查看结构化 JD。
- 触发 AI 叙事分析。
- 生成 target job 简历。

## 匹配机制

匹配分成两层：分数来自代码规则，叙事来自 AI。

### 规则匹配分

- 列表页和详情页展示的匹配分由代码规则实时计算，不持久化、不预计算。
- 输入：Profile 的技能、年限、偏好，以及结构化 JD 的必备技能、加分技能、年限要求、远程状态等字段。
- 输出：分数和等级标签（强匹配 / 可冲刺 / 需补证据），标签由分数区间映射。
- 技能匹配前先做归一化（大小写、常见别名词典），避免 "ReactJS" 与 "React" 不命中。
- 规则调整后分数在下次读取时自动生效，不需要回填数据。
- 用户没有 Profile 时不显示分数，展示引导完善 Profile 的空状态。

### AI 叙事分析

- 用户在职位详情页主动触发，产出 evidence、gaps、risks、aiNote 四类叙事；AI 不打分。
- 规则匹配分作为输入传给工作流，叙事围绕规则结果展开。
- 结果写入 `match_reports`，每用户每职位一条，重新分析时覆盖更新。表结构见《Supabase 持久化》。
- 复用与过期：报告记录生成时的 `profile_snapshot_at` 和 `job_snapshot_at`。再次查看职位时：
  - Profile 和职位都未更新：直接展示已有分析。
  - 任一已更新：仍展示旧分析，并显示"重新分析"按钮，次级描述说明 Profile 或职位已更新。
- `resume_generate` 复用最新叙事分析作为输入；没有分析或分析已过期时，引导用户先运行分析。

## Dify 工作流

职位相关 AI 能力优先通过 Dify Workflow 编排。

- `job_parse`：把管理员粘贴的 JD 文本或职位截图转成结构化职位字段（截图由 vision 模型直接解析，不做独立 OCR）。规则匹配完全依赖这些字段，解析质量直接决定匹配分可用性。
- `job_match`：基于 Profile、结构化 JD 和规则匹配结果，输出叙事分析（evidence、gaps、risks、aiNote），不输出分数。
- `resume_generate`：基于最新 match report 叙事生成 target job resume 草稿。

要求：

- 普通用户触发分析和生成，但不能修改原始职位。
- Dify 输出必须写入 `match_reports`、`ResumeVersion` 或待采纳 patch，不能只停留在 Dify 运行结果里。
- `match_reports.external_run_id` 以字符串记录 Dify `workflow_run_id`，仅用于排错；完整 AI Run Trace 体系后续接入。
- Dify 失败时职位详情仍可浏览，规则匹配分不受影响，用户可以重试分析；本地开发 fixture 只能用于验证 UI 和事件协议。

## 验收标准

- 普通用户不能上传职位。
- 管理员能创建、编辑、停用职位。
- 职位必须有导入来源。
- LinkedIn 导入失败时能降级为手动文本或截图导入。
- 列表页和详情页匹配分由规则实时计算；无 Profile 时有明确空状态。
- AI 叙事分析可复用：Profile 和职位未更新时直接展示已有报告；任一更新后展示旧报告并提示重新分析。
- AI 分析失败时职位仍可浏览、规则分仍可见，用户可重试。
