# 列表页向量匹配（Embedding 方案 B）

> 状态：**待实现**。本文是设计 spec，先不落地。
> 背景：规则匹配（`packages/domain/src/job/match.ts`）已删除——静态别名词典无法处理
> 大小写、技术写法、技能范围/层级差异，把它表达成精确百分比是过度承诺。详情页的匹配度
> 已改由 `job_match` workflow 的大模型产出（见 [工作机会](./工作机会.md)）。
> 但 AI 报告是按需、单职位、有成本的，**列表页跑不起**——所以列表页的匹配度暂时下线，
> 留待本方案补回。

## 目标

给职位列表提供一个**便宜、本地、可全量**的语义匹配信号，替代被删除的规则分。
要能处理规则匹配处理不了的情况：技术写法差异（React 18 ≈ React、K8s ≈ Kubernetes）、
技能范围与层级（「数据库优化经验」可支撑「PostgreSQL」要求）、中英混排。

## 非目标

- 不替代详情页的 AI 报告：详情页仍由 `job_match` 大模型给出匹配度 + evidence/gaps/risks。
- 不在列表页实时调用任何大模型 / embedding API（成本会被列表的 N 条职位放大）。
- 不做自训练 / 微调模型——在当前数据量和调用量下，预训练 embedding API 性价比远更高。

## 核心原则

**把成本挪到写入时，列表页只读缓存。** 匹配信号 = 两个预存向量的余弦相似度，是一次
纯数学运算（pgvector 的 `<=>` 算子），不碰任何模型。花算力的「文本 → 向量」一步
只在 JD 入库、Profile 更新时各做一次。

## 模型选型

| 维度                                  | 选择                                                            | 理由                                                      |
| ------------------------------------- | --------------------------------------------------------------- | --------------------------------------------------------- |
| 模型                                  | `text-embedding-3-small`（OpenAI，或同档多语 API，如智谱/百炼） | 多语、支持中文 + 中英混排，8191 token 输入                |
| 维度                                  | 1536（可降维到 512 省存储）                                     | pgvector 列维度须与模型一致，换模型要重算全表             |
| 为何不用 Supabase 内置 `gte-small`    | 否决                                                            | 384 维、512 token、**英文为主**，对中文 Profile/JD 不可靠 |
| 为何不自托管 HuggingFace（bge-m3 等） | 暂不                                                            | 只有当 API 费随调用量变得肉疼、且需中文原生强时才值得     |

### 约束（务必遵守）

- **Token 上限**：不要 embed 原始 JD/Profile 全文，而是 embed**蒸馏文本**——把结构化字段
  拼成短文本，既绕开长度限制、信号也更干净。
  - JD 蒸馏示例：`"前端工程师 | React, TypeScript, Next.js | 5年 | 远程"`
    （title + required/preferred skills + years + remote_status）
  - Profile 蒸馏示例：`"资深前端 | React, TypeScript, Node | 6年 | 期望远程"`
    （headline + skills + 推算年限 + 偏好）
- **归一化**：OpenAI 输出已归一化；若改自托管模型，存前要 `normalize`，否则余弦不准。
- **语言**：必须用多语模型，保证中文 Profile ↔ 英文 JD 能跨语言对上。

## 数据流

1. **JD 入库 / 更新**（`job-parse` edge function 或职位编辑保存时）
   → 生成蒸馏文本 → 调 embedding API → 写 `job_descriptions.embedding`。
2. **Profile 更新**（保存 Profile 时）
   → 生成蒸馏文本 → 调 embedding API → 写 `profiles.embedding`。
3. **列表页查询**：SQL 直接算当前用户 Profile 向量与各 JD 向量的余弦距离，返回 `match_score`，
   **不调任何模型**。

## 数据模型（pgvector）

```sql
create extension if not exists vector;

alter table job_descriptions add column embedding vector(1536);
alter table profiles         add column embedding vector(1536);

-- 可选：列表量大时加近邻索引
create index on job_descriptions using ivfflat (embedding vector_cosine_ops);
```

列表查询（`<=>` 是余弦距离，`1 - 距离` 即相似度）：

```sql
select j.id, j.title,
       round((1 - (j.embedding <=> p.embedding)) * 100)::int as match_score
from job_descriptions j
cross join (select embedding from profiles where user_id = auth.uid()) p
where j.is_active and j.embedding is not null and p.embedding is not null
order by j.embedding <=> p.embedding;
```

## 展示

- 列表页「匹配度」列展示 `match_score`（0–100）。沿用诚实表达：可只显示一个区间标签
  或弱化的数字，避免再次让用户盯着一个未校准的精确百分比。
- Profile / JD 没有向量时显示「—」或「完善 Profile」（沿用此前列表空状态）。

## 待决问题

- embedding API 供应商与 key 管理（复用现有 LLM provider 还是单独接）。
- 蒸馏文本的确切字段拼法，需与 `job_match` 大模型输入对齐，避免两套口径。
- 向量写入的触发点：在 edge function 里同步写，还是异步补算（入库先不阻塞）。
- 列表 `match_score` 的展示粒度（精确数字 vs 区间标签）。

## 落地步骤（实现时）

1. 开 `pgvector` 扩展，加 `embedding` 列 + migration。
2. 封装 embedding 调用（输入蒸馏文本 → 向量）于 shared 层。
3. JD 入库 / 编辑保存、Profile 保存处接入写向量。
4. 一次性回填存量 JD / Profile 的向量。
5. 列表页查询改为带 `match_score` 的 SQL，恢复「匹配度」列。
