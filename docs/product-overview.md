## 定位

面向开发者求职的 AI 工作台:用个人**事实库**和目标**职位描述(JD)**,生成一份可信、
可追溯、可编辑的定制简历。

## 问题

按岗位定制简历很枯燥,LLM 让它变快——但也会编造你从没有过的经历。一份读起来漂亮、却
撑不过面试的简历,比没有帮助更糟。产品存在的全部理由,就是让 AI **有用**而不让它
**说谎**。

## 核心闭环

1. **Onboarding** —— 通过上传简历(AI 解析)或手动填写来建立 profile。这个 profile 就是
   事实库。
2. **职位** —— 浏览 admin 维护的职位描述(结构化 JD,而非原始文本)。
3. **匹配** —— 按需触发,AI 分析 profile × JD,给出证据、缺口、风险和匹配分——不只是一个
   数字,而是**为什么**。
4. **生成** —— 产出一份锚定在 profile 上的 target-job 简历草稿。
5. **编辑** —— 在简历编辑器中精修;AI 对话给出 patch,由你采纳或拒绝,并留有修改日志。

## 设计原则

- **profile 是事实源。** AI 可以改写、重新组织事实的优先级;但不能编造。生成与匹配只从
  事实库读取,别无他物。
- **base 简历 vs. target-job 简历。** 你的原始简历被保留;每份定制版本都是 profile 针对
  某个 JD 的投影。

## 数据来源边界

LinkedIn 等站点有登录墙、反爬和服务条款限制,因此自动爬取**不是**依赖项。职位通过以下
方式进入系统:

- admin 粘贴 JD 文本,
- admin 粘贴链接并手动补充 JD,或
- admin 上传截图,由 Dify vision 模型解析成结构化 JD,人工确认后保存。

若将来加入浏览器辅助采集,也只读取 admin 当前可见的页面文本——不存 cookie,不绕过反爬。
